# TalentOS Track B v0.2 + Migration-Ready Track A Cores — Implementation Plan

**Target repo:** `github.com/alsaki27/talentOS-B` (clone fresh; current GitHub state = v0.1 stub scope). All work lands here. TalentOS main repo is **not touched**; pieces flagged **[MIGRATE]** are structured to lift into TalentOS later.

**Goal:** Take the v0.1 stub repo to the v0.2 state described in `HANDOVER-DEV (1).md`: working MV3 extensions, typed client, 5 ATS adapters, full copilot flow with a hard no-submit guarantee, evidence capture, a zero-dep A4 mock server, the A4 contract doc, the readiness-engine + DDL as migration-ready cores, contract tests, release-gate tools, and CI. Build green, tests green.

## Decisions locked

1. **A6 DB path (for the TalentOS side / contract):** raw-SQL Neon `query/queryOne/execute` (`@/server/db/neon.ts`). The mock server emulates this against in-memory stores shaped by the DDL; its route handlers are written raw-SQL-shaped so they lift into TalentOS `src/app/api/extension/v1/<route>/route.ts` verbatim, swapping the mock's store layer for `query/queryOne/execute`. No Supabase client in Track-A code.
2. **All work in talentOS-B repo.** Migration into TalentOS is a later, separate step; everything Track-A-side is authored to be lift-and-drop.
3. **Bundler:** esbuild per extension + copy step for `manifest.json`/`icons/` into `dist/`. `shared`/`ats-adapters`/`browser-profiles`/`readiness-engine` keep `tsc`.
4. **Migration layout:** new `packages/readiness-engine/` (pure TS, zero deps) + `sql/extension_schema.sql` at repo root.
5. **Scope naming (five `extension:*` scopes):** `extension:job:capture`, `extension:queue:read`, `extension:readiness:read`, `extension:evidence:write`, `extension:adapters:read`. One key per extension install (least privilege). Key bytes: `tos_<base64url>` (distinct from TalentOS public API `sk_live_` keys); TalentOS later adds these five scopes to `PUBLIC_API_SCOPES` and reuses `requirePublicApiScope`.

## Phase 0 — Build tooling & CI foundation (unblocks everything)

v0.1 blockers: no root `build`/`test`/`lint` scripts; no per-package tsconfig; root tsconfig points at nonexistent root `src/`; `ts-loader`/`typescript`/`webpack`/`ts-node` undeclared; no `icons/`; no `.github/workflows/`.

0.1 **`tsconfig.base.json`** at repo root: `target:ES2020`, `module:commonjs` for node pkgs, `strict:true`, `esModuleInterop`, `skipLibCheck`, `lib` per pkg. Each package adds `tsconfig.json` extending base with own `rootDir`/`outDir`/`include`.
0.2 **Root `package.json`:** add scripts `build` (ordered: shared → readiness-engine → ats-adapters → browser-profiles → extensions via esbuild glue; all controlled by `tools/build-all.mjs`), `test` (`node --test packages/*/test/` or packages that compile → `dist/test/`), `lint`, `mock-server`, `test:integration` (calls `node tools/run-integration.mjs`), `ci` (build+test+release gates). Add root devDeps hoisted to all packages: `typescript@^5.3`, `esbuild@^0.20`, `@types/chrome@^0.0.260`, `@types/node`, `eslint`, `ts-node`. Keep inter-package deps as `"*"` in each package.json (npm workspaces resolve `*` to the local workspace package — the v0.1 `"workspace:*"` was what broke `npm install`).
0.3 **esbuild glue:** `tools/build-extension.mjs <pkgName>` — bundles `src/background.ts` + injected content entry to `dist/`, then copies `manifest.json` + `icons/` to `dist/`. Each extension's `package.json` `build` calls it.
0.4 **`icons/`** per extension: committed placeholder PNGs (16/48/128) so manifests resolve.
0.5 **`.github/workflows/ci.yml`:** Node 18 (.nvmrc) — `npm ci`, `npm run build`, `npm test`, `npm run test:integration` (starts mock-server as background child process before tests, tears down after — use a helper script `tools/run-integration.mjs`), `node tools/check-no-submit.mjs`, `node tools/check-adapter-versions.mjs`.

0.6 **Test organization:** Each testable package gets a `test/` dir (e.g., `packages/readiness-engine/test/`, `packages/ats-adapters/test/`, `packages/browser-profiles/test/`). Root `npm test` runs `node --test packages/*/test/` (Node 18.17+ built-in test runner, globs `*.test.{js,mjs,cjs}`). For TS tests, compile via per-package `tsc` first (tests compile to `dist/test/`), then `node --test` targets the dist layer. Alternatively, for packages that use `tsc --noEmit` + `tsx` to run TS tests directly — implementor's choice.

0.7 **Extension communication pattern (MV3 architecture decision):** Extensions use NO `content_scripts` declaration in `manifest.json`. Instead, `background.ts` (service worker) injects content scripts dynamically via `chrome.scripting.executeScript({target:{tabId}, files:['content.js']})` when the user clicks the extension action icon (`activeTab` permission covers the current tab). ALL API calls to the TalentOS server go through the background service worker — the injected content script communicates via `chrome.runtime.sendMessage`. This avoids needing `host_permissions` in the manifest (MV3 service workers can fetch any origin; content scripts cannot). See `extension-job-capture/content.ts` as the reference pattern: DOM scraping → `chrome.runtime.sendMessage({action:'captureJob', data})` → background posts to API → `sendResponse` → content updates badge/DOM.

## Phase 1 — Migration-ready cores (A4 spec + readiness + DDL)

The three things TalentOS consumes later, authored first so everything else builds against them.

1.1 **`docs/A4_EXTENSION_API.md`** — the contract. Sections: (a) base `https://…/api/extension/v1`; (b) auth = Bearer `tos_…` or `x-api-key`, scope per route, error envelope `{error:{code,message,details?}}` on every non-2xx with the code table (`invalid_key`, `missing_scope`, `no_marker`, `malformed_marker`, `candidate_mismatch`, `queue_empty`, `not_found`, `duplicate`, `validation_error`, `adapter_expired`, `rate_limited`, `internal_error`); (c) `Idempotency-Key` header on POSTs; (d) `X-TalentOS-Client` version header; (e) the six routes (see Phase 2); (f) the readiness scoring rule; (g) versioning policy for adapters + routes (`v1` path segment, breaking changes bump to `v2`); (h) **[MIGRATE]** DDL block (copy of `sql/extension_schema.sql`).

1.2 **`packages/readiness-engine/`** — pure, zero-dep TS. Exports `computeReadinessScore({jdText, evidencedSkills, claimedSkills, knownSkillVocabulary})` → `{required, matched, missing, flagged, score, threshold}` implementing the handover arithmetic exactly:
```
required = vocabulary ∩ terms(jdText)        // normalized lowercase tokenize
matched  = required ∩ evidencedSkills       // evidencedSkills = candidate_skills with verified_by≠null
missing  = required − matched
flagged  = claimedSkills − evidencedSkills
score    = required.length ? round(100·|matched|/|required|) : 50
threshold= 70 (constant, export for tuning)
```
Pure, deterministic, no AI. **Node `--test` unit tests** incl. the incident fixture (scores <70, has a flagged unverified "kubernetes" claim). **[MIGRATE]** `src/index.ts` lifts to TalentOS `src/server/services/readinessService.ts` unchanged.

1.3 **`sql/extension_schema.sql`** — additive, idempotent (`create table if not exists`). Tables (from handover §5): `candidate_skills` (skill normalized lowercase; unique candidate_id+skill; evidence_type check; verified_by null=unverified claim), `browser_profiles` (candidate_id unique; marker_hash; decommissioned_at tombstone), `application_evidence` (application_id, screenshot_url, confirmation_scrape jsonb, idempotent), `application_outcomes` (status, source 'manual'|'cron'|'webhook'), plus **`extension_captured_jobs`** capture-staging table (apply_url unique, idempotency_key unique, promoted_job_id nullable FK→jobs — set only when a human promotes). Each table prefixed with a comment block: **[MIGRATE]** → which TalentOS migration file/Supabase migration folder to append to, and that `candidates`/`applications`/`jobs` FKs already exist (neon cons. schema confirms).

## Phase 2 — Mock A4 server (reference impl + contract tests + gates)

`tools/mock-a4-server.mjs` — zero-dep plain Node `http` server on `:4114`. Accepts any key starting with `tos_`. In-Memory Stores mirroring the DDL tables; seeds `cand_demo` (ledger: react/typescript/node.js/sql/git + unevidenced "kubernetes" claim) and `app_0001` (an approved queue ticket) — matches the handover seed contract. **Route handlers written raw-SQL-shaped** (string templates + params array, executed against the in-memory store) so they lift verbatim into TalentOS route handlers; the only swap at migration is `await mockExecute(sql, params)` → `await execute(sql, params)` from `@/server/db/neon.ts`.

Routes (the six from handover §2):
- `POST /capture-job` → validate (zod-free, manual shape checks mirroring TalentOS-side zod) title+applyUrl+jdText; dedupe by `Idempotency-Key` AND `apply_url`; insert into `extension_captured_jobs`; → `{jobId, duplicate}`.
- `GET /queue/next?candidateId=` → next human-approved application ticket with approved `profile` fill values (`applications` join `candidates`, `resume_url` per-application) → 404 `queue_empty`.
- `GET /readiness/:applicationId` → load via readiness-engine `computeReadinessScore` against store.
- `POST /readiness/preview` → same function on an arbitrary JD; **byte-identical** to the workbench path (proven by Phase 9 grep gate).
- `POST /evidence` → idempotent insert into `application_evidence`.
- `GET /adapters/manifest` → serve `packages/ats-adapters/manifest.json`.
- `/_debug/state` → dump of stores for tests/SETUP walkthroughs.

Response wrapper: `{error:{code,message,details?}}` on non-2xx; success bodies per contract. `Idempotency-Key` + `X-TalentOS-Client` headers honored.

**`test-integration/a4-contract.test.mjs`** — 10 tests via Node `--test` + plain `fetch`: auth failures, scope failures, capture+dedupe, queue empty + queue next, readiness GET + preview byte-identical, evidence idempotency, adapters manifest shape, incident fixture below threshold. Driven by env `TALENTOS_TEST_API`/`TALENTOS_TEST_KEY` (default → mock). Point at TalentOS staging later unchanged.

**`tools/run-integration.mjs`** — spawns `tools/mock-a4-server.mjs` as a child process, waits for `:4114` ready, runs `node --test test-integration/`, kills mock, exits with test code. Used by both local `npm run test:integration` and CI so there's a single lifecycle to maintain.

**Release-gate tools:**
- `tools/check-no-submit.mjs` — recursive grep of `dist/` + `src/` per extension for `\.click\(\)`/`submit()`/`dispatchEvent.*submit`/`requestSubmit`; also catches synthetic submit events. Fails on any match. Proves the copilot cannot submit.
- `tools/check-adapter-versions.mjs` — checksum of each adapter's selectors; if checksum changed but `manifest.json` `version` did not bump → fail CI. Reads committed baseline.

## Risks and edge cases to guard (implementor must address these)

1. **esbuild define for `X-TalentOS-Client`**: Each extension needs a unique client version string baked into the bundle. Pass `--define:TALENTOS_CLIENT_VERSION='"extension-job-capture/0.2.0"'` to esbuild in the per-extension build script. Do NOT rely on `process.env` at runtime (Chrome extensions have no `process`).
2. **`chrome.storage.local` vs localStorage**: MV3 service workers DO NOT have `localStorage`. ALL storage reads/writes MUST use `chrome.storage.local` (async, returns a Promise wrapped by Chrome). The v0.1 code uses synchronous `localStorage.getItem` everywhere — every call site must be updated to async.
3. **Service worker lifecycle**: MV3 service workers are short-lived (terminate after 30s idle). Any state held in module-scope variables is lost on restart. The API client's token must be fetched from `chrome.storage.local` on every API call (or cached with a short TTL). Do NOT store the token in a module variable.
4. **`chrome.scripting.executeScript` requires the `scripting` permission** (declared in manifest) AND `activeTab` (for the current non-extension URL) OR `host_permissions`. All three extensions must declare both `"scripting"` and `"activeTab"` in their manifest. Without `scripting`, `executeScript` throws.
5. **Content script injection race**: When clicking the extension icon, `chrome.action.onClicked` fires → `chrome.tabs.query` for the active tab → `chrome.scripting.executeScript`. The content script must NOT assume the DOM is fully loaded; wrap in `document.readyState === 'loading' ? document.addEventListener('DOMContentLoaded', main) : main()`.
6. **Shadow-DOM review panel isolation**: B3 (QA) and B5 (copilot) panels are injected into the ATS page via shadow DOM (`element.attachShadow({mode:'closed'})`). This prevents ATS page CSS from leaking into the panel and vice versa. Closed mode also prevents `element.shadowRoot` access from the page's JS (B5's no-submit guarantee: even if the ATS page had malicious JS, it can't reach the Submit button inside the shadow DOM).
7. **B5 fill engine must handle SPA transitions**: ATS forms often use single-page-app routing. The injected content script may need a `MutationObserver` to re-detect form presence after client-side navigation. At minimum, log a "form not found" warning when the expected adapter selectors match nothing; do NOT silently skip.
8. **Idempotency-Key generation**: Uses `crypto.randomUUID()` (available in MV3 service worker + content scripts). Sent as an `Idempotency-Key` header on all POST requests (shared client enforces this; not per-call-site).
9. **`node --test` runner on Node 18.17**: The built-in test runner is stable. Test files must use `import { describe, it, before, after } from 'node:test'; import assert from 'node:assert/strict';`. No jest/vitest globals.
10. **npm `*` deps resolve correctly**: In npm workspaces mode (`"workspaces":["packages/*"]` at root), `"@talentos/shared": "*"` in extension-job-capture resolves to the `packages/shared` package. Confirmed: this is how npm workspaces work. The v0.1 `"workspace:*"` variant was pnpm/yarn syntax that npm rejected.
